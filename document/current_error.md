## SECTION 1 — PURPOSE & SCOPE

### 1.1 Purpose
This Standard Operating Procedure (SOP) defines the requirements, methodology, and responsibilities for qualifying global technology infrastructure components used in Life Science environments. The purpose is to ensure that all infrastructure systems—servers, networks, cloud platforms, databases, and end-user computing environments—are appropriately designed, installed, operated, and maintained in a state of compliance consistent with regulatory expectations and business continuity requirements.

Qualification activities described herein ensure that infrastructure components supporting **GxP-regulated systems** are fit for intended use and that adequate evidence is generated, reviewed, and retained to support audit and inspection readiness.

### 1.2 Scope
This SOP applies to:

- **All IT infrastructure** supporting GxP applications, clinical systems, laboratory systems, and manufacturing execution systems globally.
- **Infrastructure tiers** including on-premises data centers, hybrid environments, and cloud-hosted platforms (IaaS, PaaS).
- **Organizational units** including IT, Quality Assurance (QA), Validation, and Business System Owners across all global sites.
- **New infrastructure deployments**, major upgrades, migrations, and decommissioning activities.

**Out of Scope:**
- Application-level validation (covered under application validation SOPs).
- Non-GxP infrastructure with no impact on regulated data or processes.

## SECTION 4 — QUALIFICATION LIFECYCLE FRAMEWORK

Infrastructure qualification shall follow a structured, phased lifecycle approach consistent with GAMP 5 principles and the V-Model.

### 4.1 Qualification Phases Overview

```
User Requirements Specification (URS)
          ↓
Design Qualification (DQ)
          ↓
Installation Qualification (IQ)
          ↓
Operational Qualification (OQ)
          ↓
Performance Qualification (PQ)
          ↓
Qualified State / Ongoing Monitoring
```

---

### 4.2 User Requirements Specification (URS)
- Defines the intended use, functional, performance, and compliance requirements for the infrastructure component.
- Authored by the Business/System Owner with input from IT and QA.
- Forms the baseline for all subsequent qualification phases and traceability.

### 4.3 Design Qualification (DQ)
- Verifies that the proposed infrastructure design meets the URS and applicable regulatory requirements.
- Includes architecture review, vendor assessment, and security design review.
- Deliverable: **DQ Report** with documented rationale for design decisions.

### 4.4 Installation Qualification (IQ)
- Confirms that infrastructure components are installed correctly, according to approved specifications, and in the intended environment.
- Verifies physical and logical installation, component inventory, software versioning, and configuration against baselines.
- Deliverable: **IQ Protocol & Executed Report**.

### 4.5 Operational Qualification (OQ)
- Demonstrates that the infrastructure operates as intended across defined operating ranges and boundary conditions.
- Tests include functional checks, failover/redundancy, alert mechanisms, security controls, and access management.
- Deliverable: **OQ Protocol & Executed Report**.

### 4.6 Performance Qualification (PQ)
- Demonstrates consistent performance of the infrastructure under representative real-world load and conditions over time.
- Includes performance benchmarking, availability monitoring, and end-to-end integrated testing.
- Deliverable: **PQ Protocol & Executed Report**.

### 4.7 Qualification Summary Report
- Consolidates findings across all phases (IQ/OQ/PQ).
- Documents open deviations, risk acceptances, and formal release decision.
- Requires signatures from IT Qualification Engineer, QA Lead, and Business Owner.

## SECTION 5 — INFRASTRUCTURE QUALIFICATION PROCEDURES

### 5.1 Server Infrastructure (Physical & Virtual)

**IQ Activities:**
- Verify hardware model, serial numbers, firmware versions against procurement specifications.
- Confirm rack placement, power connections, and environmental controls (temperature, humidity).
- Validate OS version, patch level, and hardening configuration against approved baseline.
- Confirm backup agent installation and storage connectivity.

**OQ Activities:**
- Verify OS boot integrity and system services startup.
- Test user authentication and role-based access control (RBAC).
- Validate audit logging (event log, syslog) generation and retention.
- Confirm patch management tool registration and vulnerability scan baseline.
- Test high availability / failover (for clustered systems).

**PQ Activities:**
- Monitor system availability and uptime over a defined period (minimum 30 days for GxP).
- Validate backup and restore procedures with documented recovery time objectives (RTO/RPO).
- Confirm sustained performance under production-representative workloads.

---

### 5.2 Network Infrastructure

**IQ Activities:**
- Verify hardware inventory (switches, routers, firewalls) against network topology diagrams.
- Confirm firmware/software versions and configuration baseline.
- Validate physical and logical network segmentation (VLANs, DMZ zones).

**OQ Activities:**
- Test network connectivity between defined infrastructure segments.
- Validate firewall ruleset against approved security policy.
- Test intrusion detection/prevention system (IDS/IPS) alert generation.
- Confirm redundant path failover and load balancing behavior.
- Validate network time protocol (NTP) synchronization across all components.

**PQ Activities:**
- Monitor network latency, packet loss, and bandwidth utilization over defined period.
- Validate that alert thresholds trigger appropriate notifications.
- Confirm audit log integrity and availability for retrospective review.

---

### 5.3 Cloud Platform Infrastructure (IaaS/PaaS)

**IQ Activities:**
- Document cloud service provider (CSP) platform, region, and service tier.
- Verify tenant configuration, subscription structure, and identity management (e.g., Azure AD, AWS IAM).
- Confirm data residency and geographic restriction settings.
- Validate CSP compliance certifications (SOC 2 Type II, ISO 27001, etc.) and shared responsibility matrix.

**OQ Activities:**
- Test identity federation and single sign-on (SSO) integration.
- Validate multi-factor authentication (MFA) enforcement.
- Test data encryption at rest and in transit configurations.
- Confirm auto-scaling rules, resource tagging policies, and activity log enablement.
- Validate disaster recovery (DR) configuration and cross-region replication.

**PQ Activities:**
- Monitor cloud resource availability, cost governance, and performance metrics.
- Execute DR runbook and validate recovery objectives are met.
- Review CSP audit reports (e.g., SOC 2) for the qualification period.

---

### 5.4 End-User Computing (EUC) Environments

**IQ Activities:**
- Verify standard hardware build, OS image version, and applied security policies (Group Policy / MDM).
- Confirm endpoint encryption (BitLocker/FileVault) and antivirus deployment.
- Validate software inventory against approved application catalog.

**OQ Activities:**
- Test user authentication, privilege restrictions, and screen lock policies.
- Validate remote management and patch deployment capability.
- Confirm data loss prevention (DLP) controls are active.

**PQ Activities:**
- Monitor patch compliance rates and vulnerability remediation timelines.
- Validate that periodic endpoint compliance scans are executed and reviewed.

## SECTION 6 — RISK MANAGEMENT & DEVIATION HANDLING

### 6.1 Risk Assessment Methodology
Risk assessments shall be conducted at the **initiation of each qualification project** and updated whenever significant changes occur. The methodology shall align with **ICH Q9** and GAMP 5 risk-based thinking.

**Risk Assessment Steps:**
1. **Identify** infrastructure components and their GxP impact.
2. **Assess** probability, severity, and detectability of potential failures using a Risk Priority Number (RPN) matrix.
3. **Classify** risk as Low / Medium / High based on agreed thresholds.
4. **Define** mitigation controls and assign ownership.
5. **Document** residual risk and obtain QA acceptance.

**Risk Matrix:**

| Severity | Probability: Low | Probability: Medium | Probability: High |
|---|---|---|---|
| **High** | Medium | High | High |
| **Medium** | Low | Medium | High |
| **Low** | Low | Low | Medium |

---

### 6.2 Deviation Management

All deviations identified during qualification execution shall be managed as follows:

| Step | Action |
|---|---|
| **Detection** | Tester documents deviation in the protocol with timestamp, description, and evidence. |
| **Classification** | Deviation classified as Critical, Major, or Minor by QA. |
| **Investigation** | Root cause analysis (RCA) performed by responsible engineer within defined timeframe. |
| **Impact Assessment** | Determine if deviation impacts GxP data integrity, system performance, or compliance posture. |
| **Resolution** | Corrective action implemented, retested, and re-documented. |
| **CAPA** | For Critical/Major deviations, a formal Corrective and Preventive Action (CAPA) is raised. |
| **Closure** | QA reviews and approves deviation closure before qualification report sign-off. |

**Deviation Classification:**
- **Critical:** Directly impacts data integrity, patient safety, or product quality.
- **Major:** Impacts system function but can be mitigated; requires CAPA.
- **Minor:** Low impact; can be documented and accepted with documented rationale.

## SECTION 8 — CHANGE CONTROL & PERIODIC REVIEW

### 8.1 Change Control for Qualified Infrastructure

Any change to a qualified infrastructure component must be assessed for its **impact on the qualified state** prior to implementation.

**Change Control Process:**

```
Change Request Initiated
        ↓
IT Qualification Engineer performs Impact Assessment
        ↓
QA reviews and classifies change (Major / Minor / Emergency)
        ↓
Change Control Board (CCB) approval
        ↓
Change implemented by System Administrator
        ↓
Requalification activities executed (scope based on impact assessment)
        ↓
Qualification documentation updated
        ↓
QA approves return to qualified state
```

**Change Classification:**

| Classification | Description | Requalification Required? |
|---|---|---|
| **Major** | Significant configuration change, OS upgrade, platform migration, new component addition | Yes — partial or full requalification |
| **Minor** | Patch updates within approved baseline, parameter adjustments within validated range | Partial OQ / documented assessment |
| **Emergency** | Unplanned critical fixes (e.g., security patches) | Retrospective assessment within 30 days |

Qualified infrastructure shall be subject to **formal Periodic Review** on an annual cycle (or more frequently if risk warrants).

**Periodic Review Activities:**

| Activity | Description |
|---|---|

| Patch & Vulnerability Review | Confirm all security patches are current; review open vulnerability findings. |
| Audit Trail Review | Sample review of audit trail entries for completeness and integrity. |
| Incident & Deviation Review | Summarize and trend incidents and deviations since last review. |
| Access Control Review | Validate that user access rights remain appropriate and authorized. |
| Change History Review | Confirm all changes were processed through approved change control. |
| Risk Re-assessment | Update risk assessment if significant changes have occurred. |
| Document Refresh | Update qualification documentation to reflect current state. |

**Periodic Review Output:**
- A **Periodic Review Report** signed by the IT Qualification Engineer, System Administrator, and QA.
- Open actions tracked in the CAPA system.
- Confirmation that the system remains in a qualified state or initiation of requalification if required.
